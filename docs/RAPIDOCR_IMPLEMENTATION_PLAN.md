# RapidOCR 简历关键词检测实施方案

文档日期：2026-06-30

项目：BOSS 直聘自动刷简历工具

当前阶段：技术方案设计，暂不修改代码

## 一、实施目标

使用 RapidOCR 识别屏幕上当前可见的候选人简历文字，替换现有的网页全选复制方案。

新方案只分析屏幕截图，不读取浏览器 DOM、不调用 BOSS 接口，也不使用 WebDriver。核心目标是避免 `Ctrl+A` 同时复制全屏详情覆盖层和底层推荐页文本，从而防止其他候选人的关键词触发当前候选人的邮件转发。

本阶段采用以下约束：

- OCR 引擎使用 RapidOCR；
- 完全在本机运行；
- 使用固定最大扫描次数判断扫描结束；
- 关键词只做标准化后的精确匹配；
- 暂不实现模糊匹配；
- 暂不修改现有代码。

## 二、RapidOCR 是否需要 AI 大模型 API

不需要。

RapidOCR 是本地 OCR 推理工具，可以通过 ONNX Runtime 在电脑 CPU 上运行检测、文字方向分类和文字识别模型。模型文件下载并部署到本机后，截图和识别结果不需要上传给第三方大模型服务。

它与调用云端大模型 API 的区别如下：

| 项目 | RapidOCR 本地方案 | 云端大模型/OCR API |
| --- | --- | --- |
| 是否联网识别 | 否 | 通常需要 |
| 是否需要 API Key | 否 | 通常需要 |
| 是否按调用次数收费 | 否 | 通常会收费 |
| 简历图片是否上传 | 否 | 通常会上传 |
| 推理算力来源 | 本机 CPU/GPU | 服务提供方 |
| 数据隐私 | 数据留在本机 | 取决于服务条款 |

RapidOCR 本身不是聊天式大语言模型。它使用专门的文字检测和文字识别模型完成 OCR，不需要接入 OpenAI、通义千问、文心一言或其他大模型 API。

参考：[RapidOCR 官方 GitHub 仓库](https://github.com/RapidAI/RapidOCR)。

## 三、是否需要持续投入

### 1. 不存在按次调用费用

使用本地 RapidOCR 不需要购买识别次数，也没有每月 API 账单。模型推理消耗的是本机 CPU、内存和少量电力。

### 2. 可能存在的实际投入

持续投入主要来自工程维护，而不是服务费用：

- BOSS 页面布局变化后，可能需要重新校准截图区域；
- 浏览器缩放或 Windows 显示缩放变化后，可能需要调整截图参数；
- OCR 模型或依赖升级时，需要重新测试识别效果；
- 误命中或漏命中时，需要根据命中截图调整图像预处理；
- OCR 截图和日志需要定期清理，避免存储候选人个人信息；
- 如果未来要求更快的识别速度，可能需要升级 CPU 或选择硬件加速运行库。

### 3. 初次部署的网络需求

首次安装 Python 依赖、OCR 运行库和模型文件时通常需要联网下载。完成安装后，可以离线运行。

如果打包成独立 EXE，模型文件应随程序一起分发或在首次安装时下载，并校验模型是否完整。

## 四、电脑配置要求

RapidOCR 不强制要求独立显卡。对于本项目每次只识别一张网页截图的场景，CPU 推理即可。

### 最低可用配置

- Windows 10/11 64 位；
- 4 核 CPU；
- 4 GB 可用内存；
- 约 1 GB 可用磁盘空间，用于依赖、模型、日志和临时截图；
- 1920 × 1080 或其他固定分辨率显示器。

最低配置可以运行，但 OCR 速度可能偏慢，不建议同时运行大量占用 CPU 或内存的软件。

### 推荐配置

- Windows 10/11 64 位；
- 近几年推出的 Intel Core i5、AMD Ryzen 5 或同等级 CPU；
- 8 GB 以上内存；
- SSD；
- 固定显示分辨率、Windows 缩放和浏览器缩放；
- 不要求独立显卡。

### 性能说明

OCR 速度主要受以下因素影响：

- 截图分辨率和裁剪面积；
- 页面文字数量；
- CPU 单核和多核性能；
- 使用的检测、分类和识别模型大小；
- 是否每次都重新加载模型；
- 是否进行图像放大、锐化等预处理。

程序必须在启动时初始化一次 OCR 引擎，然后在整个运行期间复用。不能每扫描一屏就重新加载模型，否则速度会明显下降。

4 GB 和 8 GB 是本项目的工程估算，不是 RapidOCR 官方规定的硬性门槛。正式实施前需要在实际运行 BOSS 的目标电脑上做性能测试。

## 五、目标运行流程

修改后的候选人处理流程计划如下：

```text
打开当前候选人详情
        ↓
点击详情中心，确保滚轮焦点正确
        ↓
截取候选人详情可见区域
        ↓
RapidOCR 识别当前截图
        ↓
标准化 OCR 文本并精确匹配关键词
        ↓
命中 ──→ 二次截图复核 ──→ 邮件转发
        │
        未命中
        ↓
向下滚动固定距离
        ↓
重复截图，直到达到最大扫描次数
        ↓
未命中则结束当前候选人扫描
        ↓
完成剩余停留时间并切换下一位
```

## 六、第一阶段：准备真实截图样本

在修改程序之前，先手动收集一组 BOSS 候选人详情截图。

样本应覆盖：

- 短简历；
- 长简历；
- 中英文混排；
- 包含数字、学校和公司名称；
- 页面顶部、中部和底部；
- 有作品集和无作品集；
- 不同长度的工作经历和教育经历；
- 含目标关键词和不含目标关键词；
- 浏览器缩放及 Windows 缩放与正式运行环境一致。

建议至少准备 20–30 张截图，其中一半含关键词、一半不含关键词。

样本必须保存在 Git 忽略目录中，并在测试结束后妥善清理，避免泄露候选人个人信息。

## 七、第二阶段：确定截图区域

### 1. 只截取网页内容

截图区域需要排除：

- 浏览器标签栏和地址栏；
- Windows 任务栏；
- 页面固定顶部导航；
- 与简历无关的操作按钮；
- 可能出现的邮件转发弹窗；
- 程序控制台窗口。

应尽量保留当前候选人简历的正文区域。

### 2. 固定运行环境

正式运行前应固定：

- 屏幕分辨率；
- Windows 显示缩放比例；
- Edge 窗口位置和大小；
- Edge 页面缩放比例；
- BOSS 详情页布局。

截图区域可以在配置区使用以下形式管理：

```text
OCR_REGION_LEFT
OCR_REGION_TOP
OCR_REGION_WIDTH
OCR_REGION_HEIGHT
```

具体数值必须根据目标 Windows 电脑截图校准，不能直接在当前 macOS 开发环境中猜测。

## 八、第三阶段：接入 RapidOCR

正式编码时计划完成以下工作：

1. 在依赖文件中加入 RapidOCR 及对应 ONNX Runtime 运行库；
2. 在程序启动阶段加载 OCR 模型；
3. 如果模型加载失败，关闭自动转发并明确记录错误；
4. 使用现有 `mss` 对指定屏幕区域截图；
5. 将截图转换为 RapidOCR 可接受的图像格式；
6. 调用 OCR，取得文字、置信度和文字框坐标；
7. 按文字框的纵向、横向位置恢复基本阅读顺序；
8. 将结果交给关键词匹配模块；
9. 整个运行期间复用同一个 OCR 实例。

OCR 初始化失败时不能退回现有的全页剪贴板检测，因为这会重新引入已确认的错误转发风险。

## 九、第四阶段：文字标准化和精准匹配

本阶段只实现精准命中。

### 1. 允许的文本标准化

为了处理 OCR 分行和中英文排版，可以进行不会改变文字含义的标准化：

- 英文字母统一为小写；
- 全角字符转换为半角；
- 删除换行符；
- 删除中文字符之间的无意义空格；
- 合并连续空白字符；
- 保留汉字、字母和数字本身；
- 对关键词执行相同的标准化。

例如：

```text
广东工贸职业
技术学院
```

标准化为：

```text
广东工贸职业技术学院
```

### 2. 精准命中规则

只有标准化后的 OCR 文本完整包含标准化关键词时，才算命中。

```text
normalized_keyword in normalized_ocr_text
```

不允许以下行为：

- 编辑距离模糊匹配；
- 拼音相似匹配；
- 同义词推断；
- 大模型语义判断；
- OCR 相似字自动替换；
- 只命中部分关键词就转发。

### 3. OCR 置信度

识别结果需要记录置信度。初期可设置一个保守的最低置信度，低于阈值的文字不参与自动转发，或者只写入调试日志。

具体阈值不能提前武断确定，需要用真实截图测试误识别情况后再决定。

## 十、第五阶段：固定最大扫描次数

本项目采用固定最大扫描次数，不实现复杂的页面底部识别。

### 推荐流程

```text
MAX_OCR_SCANS_PER_CANDIDATE = 8

第 1 次：打开详情后的首屏
第 2 次：向下滚动后截图
第 3 次：继续向下滚动后截图
...
第 8 次：最后一次截图
```

如果任意一次精确命中关键词，立即停止后续扫描并进入复核流程。如果 8 次均未命中，则判定当前候选人未命中。

最大扫描次数应当通过真实简历样本确定。建议初始测试范围为 6–10 次，再根据长简历的覆盖情况选择最终值。

### 滚动距离

每次建议向下滚动约半屏到三分之二屏，并保留一定重叠：

- 避免关键词刚好跨越两屏而被漏掉；
- 页面渲染稍有偏移时仍能覆盖全部内容；
- 不需要对相邻截图做复杂拼接和去重。

滚动距离和间隔可以小幅随机化，但扫描方向应始终向下，不能随机上下切换。

### 扫描耗时

扫描时间应计入原本每位候选人的 12–18 秒停留时间，避免“OCR 扫描完成后再额外停留 12–18 秒”导致总耗时翻倍。

## 十一、第六阶段：命中后二次确认

为降低误转发风险，第一次命中后不应立即点击转发。

建议执行：

1. 等待页面稳定约 0.5–1 秒；
2. 对相同屏幕区域重新截图；
3. 再次进行 OCR；
4. 对同一关键词执行精准匹配；
5. 两次都命中才允许触发转发。

二次确认仍只使用本地 OCR，不使用模糊匹配或大模型。

如果第二次未命中，应记录为“疑似命中”，但不转发。

## 十二、第七阶段：日志和证据

建议为每次 OCR 扫描记录：

- 当前批次序号；
- 第几次扫描；
- OCR 耗时；
- 识别文字数量；
- 标准化后的文本长度；
- 是否命中；
- 命中的关键词；
- 命中文字的置信度；
- 是否通过二次确认。

命中时保存当前裁剪区域截图和 OCR 文本，例如：

```text
logs/ocr_hits/20260630_190501_scan03.png
logs/ocr_hits/20260630_190501_scan03.txt
```

未命中的全部截图不建议长期保存，否则会迅速占用磁盘并增加个人信息风险。调试阶段可以短期保存，正式运行时默认关闭。

`logs/` 已被 Git 忽略，但仍应设置人工或自动清理周期。

## 十三、第八阶段：测试与验收

### 离线截图测试

在真实网页运行前，先使用已保存截图验证：

- 含关键词截图能够精确命中；
- 不含关键词截图不会命中；
- 分成两行的关键词可以在标准化后命中；
- 相似但不相同的文字不会命中；
- 中英文大小写能够按规则处理；
- 低置信度错误文字不会触发转发。

### 页面运行测试

建议依次测试：

1. 关闭邮件转发，只记录 OCR 结果；
2. 连续浏览 20–50 位候选人，人工核对命中情况；
3. 验证固定扫描次数是否覆盖长简历；
4. 验证 OCR 和停留时间的总耗时；
5. 验证滚动后右方向键焦点是否正常；
6. 使用测试邮箱启用转发；
7. 确认二次 OCR 通过后才发送；
8. 最后再进入正式运行。

### 验收标准

- 底层推荐页含关键词、当前详情不含时，不得转发；
- 当前详情含关键词时，应在对应屏幕位置命中；
- 同一个精确关键词需要连续两次 OCR 确认；
- OCR 失败时不得转发；
- 达到最大扫描次数后必须结束扫描，不得无限滚动；
- OCR 扫描不会明显破坏原有暂停、停止和翻页操作；
- 所有命中都能通过日志和局部截图复查。

## 十四、模糊匹配后续需求

模糊匹配保留为后续需求，本阶段不实施。

未来如需实现，应单独评估：

- OCR 常见中文相似字；
- 编辑距离阈值；
- 关键词长度与误报率；
- 学校、公司和技能词的不同容错策略；
- 模糊命中是否只提醒人工确认，而不自动转发；
- 是否需要建立关键词白名单和别名表。

在没有足够真实样本和误报统计前，不建议让模糊匹配直接触发邮件转发。

## 十五、建议实施顺序

建议按以下顺序推进：

1. 收集并脱敏保存真实截图样本；
2. 在独立测试脚本中验证 RapidOCR 中文识别效果；
3. 确定截图区域、浏览器缩放和显示缩放；
4. 确定最大扫描次数与单次滚动距离；
5. 实现文字排序、标准化和精准匹配；
6. 实现命中后二次 OCR；
7. 增加命中截图和审计日志；
8. 接入现有浏览循环，但先禁用真实邮件转发；
9. 完成 20–50 位候选人的人工对照测试；
10. 使用测试邮箱小规模验证；
11. 达到验收标准后再启用正式转发；
12. 最后再评估模糊匹配需求。

## 十六、最终建议

RapidOCR 适合当前项目的原因是：它完全在本机识别屏幕上真实可见的简历文字，不需要大模型 API，也不需要持续购买调用额度，同时不改变现有以鼠标、键盘和屏幕坐标为核心的自动化架构。

第一版应保持保守：固定次数扫描、精确匹配、两次 OCR 确认、识别失败不转发。这个策略会容忍少量漏检，但优先避免把不符合关键词的候选人错误转发。
