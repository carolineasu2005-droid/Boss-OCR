# RapidOCR 关键词检测改造详细实施文档

文档日期：2026-06-30

状态：实施设计，暂不修改代码

适用项目：BOSS 直聘自动刷简历工具

## 一、改造目标

本次改造只替换关键词检测链路：

```text
原方案：点击详情页 → Ctrl+A → Ctrl+C → 搜索整个网页文本

新方案：截取屏幕可见详情区域 → RapidOCR → 标准化文字 → 精确匹配
```

以下流程不得改变：

- 用户输入关键词和备用邮箱；
- 浏览器窗口置顶；
- 点击第一位候选人；
- 每位候选人的停留时间；
- 鼠标滚动行为；
- 右方向键切换候选人；
- 邮件转发窗口的点击坐标和操作顺序；
- 最近联系邮箱和备用邮箱逻辑；
- 连续转发限制；
- 每 100 位候选人按 `F5` 刷新；
- `Esc` 停止和空格暂停；
- 现有日志主流程。

允许新增的内容仅限于：

- RapidOCR 初始化；
- OCR 截图区域校准；
- 屏幕区域截图；
- OCR 文字识别；
- 文字标准化；
- 精确关键词匹配；
- 固定次数分屏扫描；
- 命中后二次 OCR 确认；
- OCR 专用日志和调试证据。

## 二、核心设计原则

### 1. 不再使用剪贴板检测简历关键词

关键词检测不得继续调用以下旧链路：

```text
Ctrl+A
Ctrl+C
get_clipboard_text()
```

邮箱输入框检测仍然可以保留现有剪贴板逻辑，因为该逻辑属于邮件转发环节，不在本次改造范围内。

### 2. OCR 只分析屏幕上真实可见的像素

截图必须来自操作系统屏幕区域，例如使用 `mss` 截取指定矩形。

不得使用：

- 浏览器整页截图；
- DOM 文本；
- WebDriver；
- 浏览器调试协议；
- BOSS 内部接口；
- JavaScript 注入。

### 3. 识别失败时禁止转发

如果出现以下任一情况，当前候选人应判定为“无法确认”，不得触发邮件转发：

- OCR 引擎未初始化；
- 截图区域未设置；
- 截图失败；
- OCR 调用异常；
- OCR 没有返回有效文字；
- 首次命中但二次确认失败。

不得自动退回旧的全页复制方案。

## 三、启动时手动拖框校准是否可行

可行，且建议实施。

程序可以显示一个半透明全屏选框，用户按住鼠标左键，从候选人详情区域左上角拖到右下角。松开鼠标后，程序记录：

```text
left
top
width
height
```

后续所有 OCR 截图都使用这个区域。

这样不需要提前知道：

- 公司电脑分辨率；
- Windows 显示缩放；
- Edge 窗口尺寸；
- Edge 窗口位置；
- 候选人详情的固定坐标。

但用户仍需保证运行过程中不移动或缩放 Edge 窗口，否则已选择的区域会失效。

## 四、校准界面的推荐实现

### 1. 使用独立的轻量选框窗口

推荐使用 Python 标准库 `tkinter` 创建选框，不需要引入大型 GUI 框架。

选框窗口应具备：

- 覆盖当前显示器；
- 无标题栏；
- 置顶显示；
- 半透明背景；
- 鼠标拖动时绘制矩形边框；
- 显示选择区域的宽度和高度；
- 松开左键完成选择；
- `Esc` 取消选择并停止自动转发；
- 支持重新选择；
- 选择完成后立即关闭，不留在屏幕上。

### 2. 推荐的用户提示

弹出选框前，控制台显示：

```text
请框选候选人详情正文区域：
1. 从详情区域左上角按住鼠标左键
2. 拖动到右下角后松开
3. 排除浏览器地址栏、任务栏和固定按钮
4. 按 ESC 取消
```

### 3. 应该框选什么

用户应框选当前候选人简历的主要可见内容区域，包括：

- 候选人基本信息；
- 个人优势；
- 工作经历；
- 项目经历；
- 教育经历；
- 随页面滚动出现的正文区域。

尽量排除：

- 浏览器标签栏和地址栏；
- Windows 任务栏；
- 页面固定导航；
- 右上角操作图标；
- 底部固定按钮；
- 滚动条；
- 控制台窗口；
- 邮件转发弹窗。

不需要紧贴文字边缘，四周可以保留少量空白。

## 五、校准触发时机

为了不改动现有启动和转发流程，建议采用“首次检测时延迟校准”。

### 推荐调用顺序

```text
程序正常启动
        ↓
输入关键词和备用邮箱
        ↓
程序点击第一位候选人
        ↓
第一次调用 detect_keywords()
        ↓
发现 OCR 区域尚未设置
        ↓
弹出手动拖框
        ↓
用户框选当前候选人详情区域
        ↓
保存本次运行的 OCR 区域
        ↓
开始第一次 OCR 检测
```

这种方式只需要在关键词检测模块内部加入：

```text
ensure_ocr_region_calibrated()
```

邮件转发函数和其他自动化流程不需要知道校准界面的存在。

### 为什么不在程序刚启动时立即校准

程序刚启动时，候选人详情可能尚未打开。用户无法准确判断正文区域。因此，应当等第一位候选人详情已经显示后再弹出选框。

## 六、校准结果的保存策略

### 第一版建议：只在本次运行中保存

第一版将区域保存在内存中：

```text
ocr_region = {
    left,
    top,
    width,
    height
}
```

程序重启后重新框选。

优点：

- 不会使用过期坐标；
- 电脑分辨率或窗口位置变化时不会误用旧设置；
- 实现简单；
- 更符合“不需要预先校准”的目标。

### 后续可选：保存并允许重新校准

未来可以把区域保存到本地配置文件，并提供：

- 使用上次区域；
- 重新框选；
- 截图预览；
- 删除旧配置。

本阶段不需要实施持久化。

## 七、显示缩放与坐标换算

这是拖框功能必须认真处理的技术点。

`tkinter` 返回的可能是界面逻辑坐标，而 `mss` 使用的通常是屏幕实际像素坐标。在以下环境中可能出现比例差异：

- Windows 显示缩放为 125%、150% 或 200%；
- Mac Retina 显示器；
- 多显示器具有不同缩放比例；
- Edge 位于非主显示器。

实施时必须验证：

```text
拖框坐标
        ↓
mss 截图
        ↓
保存校准预览图
        ↓
预览图是否与用户框选范围一致
```

不能仅凭拖框矩形看起来正确就认为坐标正确。

第一版建议只支持主显示器，并明确要求 Edge 位于主显示器。多显示器支持可以在基础功能稳定后增加。

## 八、校准后的预览确认

用户松开鼠标后，程序应立即使用所选区域截取一张预览图。

推荐确认流程：

```text
区域已选择：left=..., top=..., width=..., height=...
已保存预览：logs/ocr_calibration_preview.png

按 Enter 确认
按 R 重新框选
按 Esc 取消
```

如果不希望用户打开图片查看，后续可以增加一个简单预览窗口；第一版使用保存图片加键盘确认即可。

校准预览属于候选人个人信息，必须保存在 `logs/` 下，不得提交到 Git，并应在下一次校准时覆盖旧文件。

## 九、模块划分

建议新增独立 OCR 模块，避免把所有逻辑继续堆入 `simple_brush.py`。

计划结构：

```text
simple_brush.py
ocr_detector.py
ocr_calibration.py
ocr_text.py
```

### `ocr_calibration.py`

职责：

- 显示全屏拖框；
- 记录鼠标起点和终点；
- 计算矩形区域；
- 校验最小宽度和高度；
- 保存校准预览；
- 返回标准化区域配置。

### `ocr_detector.py`

职责：

- 初始化并复用 RapidOCR；
- 使用 `mss` 截取区域；
- 调用 OCR；
- 整理文字和置信度；
- 执行单屏识别；
- 执行固定次数扫描；
- 执行命中后二次确认；
- 返回结构化检测结果。

### `ocr_text.py`

职责：

- 按文字框位置排列文字；
- 文本标准化；
- 关键词标准化；
- 精确匹配；
- 保留未来模糊匹配接口，但不实现模糊命中。

### `simple_brush.py`

只需要：

- 启动时准备 OCR 检测器；
- 在原 `detect_keywords()` 位置调用 OCR 检测器；
- 根据检测结果决定是否调用原转发函数；
- 保留现有其他业务流程。

## 十、RapidOCR 初始化步骤

### 1. 安装依赖

后续编码时需要在 `requirements.txt` 中增加：

- RapidOCR 主包；
- ONNX Runtime CPU 运行库；
- RapidOCR 所需图像依赖。

具体包名和兼容版本应在实际开发时依据 RapidOCR 官方文档锁定，并分别验证 macOS 和 Windows。

### 2. 只初始化一次

OCR 引擎应在程序运行期间只初始化一次：

```text
程序启动
    ↓
创建 RapidOCR 实例
    ↓
处理全部候选人时复用
```

不得每扫描一屏重新加载模型。

### 3. 初始化失败的处理

如果 RapidOCR 初始化失败：

- 写入明确错误日志；
- 禁用自动转发；
- 可以继续浏览候选人；
- 不得退回全页复制检测；
- 不得因为 OCR 不可用而默认命中。

## 十一、单屏截图识别流程

每次单屏检测执行：

1. 确认 OCR 区域已经校准；
2. 等待页面滚动或加载稳定；
3. 使用 `mss` 截取指定区域；
4. 将截图转换为 OCR 所需格式；
5. 调用 RapidOCR；
6. 获取每个文字框、文字内容和置信度；
7. 按文字框从上到下、从左到右排序；
8. 拼接为一段可搜索文本；
9. 标准化 OCR 文本；
10. 逐个精确匹配关键词；
11. 返回命中关键词、置信度和调试信息。

建议使用结构化返回值：

```text
success
matched
matched_keyword
normalized_text
raw_items
average_confidence
elapsed_seconds
error
```

## 十二、文字标准化规则

本阶段只允许不会改变语义的标准化：

- 英文字母转小写；
- 全角字符转半角；
- 删除换行；
- 合并连续空格；
- 删除中文字符之间由 OCR 产生的空格；
- 对关键词执行完全相同的处理。

例如：

```text
OCR：广东工贸职业
     技术学院

标准化：广东工贸职业技术学院
```

精确匹配规则：

```text
normalized_keyword in normalized_ocr_text
```

本阶段不做：

- 编辑距离；
- 相似字替换；
- 拼音匹配；
- 语义匹配；
- 同义词推断；
- 大模型判断。

模糊匹配只保留未来扩展位置。

## 十三、固定最大扫描次数

不实现页面底部视觉判断，采用固定最大扫描次数。

初始建议：

```text
MAX_OCR_SCANS_PER_CANDIDATE = 8
```

单个候选人的扫描流程：

```text
第 1 屏截图并 OCR
        ↓ 未命中
向下滚动约半屏
        ↓
第 2 屏截图并 OCR
        ↓ 未命中
继续滚动
        ↓
最多执行到第 8 屏
```

任意一屏命中后立即停止继续向下扫描，进入二次确认。

8 次只是初始值，最终应根据实际长简历测试决定。建议测试 6、8、10 次的覆盖率和耗时。

## 十四、滚动策略

OCR 扫描阶段不能继续使用现有的随机上下滚动逻辑，否则可能漏掉内容。

OCR 扫描内部应使用：

- 始终向下；
- 每次约半屏到三分之二屏；
- 相邻画面保留重叠；
- 滚动距离可以轻微随机；
- 滚动后等待页面稳定再截图。

OCR 扫描完成后，原有停留和浏览逻辑保持不变。

需要注意：扫描行为会占用原本的停留时间。实施时应避免扫描 8 屏后再额外完整等待 12–18 秒，否则每位候选人的总耗时会大幅增加。

## 十五、命中后二次确认

第一次识别到关键词后：

1. 不立即调用邮件转发；
2. 保持当前滚动位置；
3. 等待 0.5–1 秒；
4. 对同一区域再次截图；
5. 再次调用 RapidOCR；
6. 再次执行完全相同的精确匹配；
7. 两次均命中同一关键词，才返回 `matched=True`。

如果第二次未命中：

- 返回未确认；
- 不触发转发；
- 保存调试日志；
- 可保存两次截图供人工分析。

## 十六、与现有代码的接口边界

原流程大致为：

```text
keyword_hit = detect_keywords()

if keyword_hit:
    forward_one_candidate()
```

改造后仍应保持这个业务接口，或者使用兼容的结构化结果：

```text
detection = detect_keywords_with_ocr()

if detection.confirmed_match:
    forward_one_candidate()
```

`forward_one_candidate()` 本身不修改。

以下函数原则上不修改：

- `forward_one_candidate()`；
- 邮箱输入和验证代码；
- `next_candidate()`；
- `refresh_page()`；
- `click_first_candidate()`；
- 暂停和停止监听；
- 用户输入流程。

如果为了调用 OCR 校准必须在主流程增加一行初始化或首次调用，应保持为最小接入，不改变原行为。

## 十七、Mac 开发步骤

私人 Mac 可以先完成：

1. 建立 OCR 独立模块；
2. 安装并验证 RapidOCR CPU 推理；
3. 用本地图片测试中文识别；
4. 实现 `tkinter` 拖框；
5. 处理 Retina 逻辑坐标与实际像素坐标；
6. 用 `mss` 截取拖框区域；
7. 保存并检查校准预览；
8. 实现 OCR 结果排序；
9. 实现文本标准化；
10. 实现精确关键词匹配；
11. 实现二次确认；
12. 用一组静态图片模拟最多 8 次滚动扫描；
13. 编写单元测试；
14. 测量单屏 OCR 耗时；
15. 模拟 OCR 失败和无文字场景。

Mac 开发期间不需要运行现有 Windows 专用的 `simple_brush.py` 全流程。OCR 模块应能通过独立测试入口读取截图。

## 十八、Windows 最终接入步骤

Mac 端 OCR 核心稳定后，在公司 Windows 电脑完成：

1. 安装 Windows 版 OCR 依赖；
2. 打开真实 BOSS 候选人详情；
3. 验证全屏拖框；
4. 验证 Windows DPI 缩放坐标；
5. 检查校准预览是否准确；
6. 测试真实字体和页面背景下的 OCR；
7. 确定每次滚动距离；
8. 确定最大扫描次数；
9. 先关闭真实邮件转发，只记录匹配结果；
10. 人工对照至少 20–50 位候选人；
11. 使用测试邮箱验证少量转发；
12. 最后再启用正式转发。

logs/ocr.log

日志记录：

- 校准区域；

## 二十、测试用例

### 校准测试

- 正常从左上拖到右下；
- 从右下反向拖到左上；
- 选择区域过小；
- 按 `Esc` 取消；
- 重新框选；
- Windows 125% 和 150% 缩放；
- Mac Retina；
- Edge 位于主显示器；
- 选框完成后的截图与矩形一致。

### OCR 测试

- 中文学校名称；
- 中文公司名称；
- 中英文技能名称；
- 关键词跨两行；
- 关键词在首屏；
- 关键词在第 8 屏；
- 完全不含关键词；
- 相似但不相同的词；
- 空白截图；
- 页面加载中的截图；
- OCR 调用异常。

### 流程测试

- 第一次命中、第二次确认命中：允许转发；
- 第一次命中、第二次未命中：禁止转发；
- 8 屏均未命中：正常进入下一位；
- OCR 初始化失败：继续浏览但禁用转发；
- 暂停发生在 OCR 扫描期间；
- `Esc` 发生在校准期间；
- 刷新后继续使用本次运行的区域；
- 邮件转发流程与改造前完全一致。

## 二十一、验收标准

满足以下条件才算完成：

- 用户可以在第一位候选人详情显示后手动拖框；
- 不需要预先填写截图坐标；
- 校准预览与用户选择区域一致；
- 关键词检测不再调用网页全选复制；
- OCR 只识别所选可见区域；
- 最多执行固定次数扫描；
- 只进行标准化后的精确匹配；
- 命中必须通过二次 OCR；
- OCR 失败时不转发；
- 邮件转发函数没有修改；
- 邮箱填写逻辑没有修改；
- 翻页、刷新、停留、暂停和停止逻辑保持原行为；
- Mac 可独立测试 OCR 核心；
- Windows 完成 DPI、截图范围和真实页面验收。

## 二十二、实施顺序

建议严格按以下顺序执行：

1. 在 Mac 建立跨平台 OCR 模块；
2. 实现静态图片 OCR；
3. 实现文本排序与精准匹配；
4. 实现单屏二次确认；
5. 实现固定 8 次图片序列扫描；
6. 实现 `tkinter` 手动拖框；
7. 实现 `mss` 局部截图；
8. 处理 Retina 和 DPI 坐标换算；
9. 增加校准预览与日志；
10. 编写离线单元测试；
11. 将 OCR 检测以最小接口接入 `simple_brush.py`；
12. 保持邮件转发和其他流程代码不变；
13. 在 Windows 实际页面上校准和验证；
14. 先进行只检测、不转发测试；
15. 再使用测试邮箱小规模验证；
16. 达到验收标准后投入正式使用。

## 二十三、最终设计决策

本次采用以下正式方向：

```text
首次关键词检测时手动拖框
+ mss 局部屏幕截图
+ RapidOCR 本地 CPU 识别
+ 固定最大扫描次数
+ 标准化后的精确匹配
+ 同屏二次 OCR 确认
+ OCR 失败不转发
```

所有新增能力封装在关键词检测模块中。邮件转发及其他自动浏览环节保持现状，避免扩大改造范围和引入新的操作风险。
