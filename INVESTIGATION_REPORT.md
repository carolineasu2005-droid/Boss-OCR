# 关键词连续误触发问题调查报告（修订版）

调查日期：2026-06-30

调查范围：`simple_brush.py`、`logs/simple_brush.log`、`logs/clipboard_dump.txt`

限制条件：本次只分析代码和历史运行数据，未修改程序代码。

## 一、最终结论

现场信息证明候选人可以正常切换后，重新检查完整剪贴板数据，已经找到确定的程序内部漏洞：

> `detect_keywords()` 通过 `Ctrl+A`、`Ctrl+C` 复制的是整个 BOSS 网页文档，而不是当前候选人的详情。虽然屏幕上只有全窗口候选人详情，但复制结果还包含被全屏详情覆盖的底层推荐页中多位候选人的摘要。只要底层页面仍保留含关键词的候选人，后续无论当前打开哪份简历，全文搜索都仍会命中并触发转发。

因此，这不是 `keyword_hit` 没有清零，也不以“页面没有翻到下一位”为必要条件。根因是关键词检测的数据范围错误。

结论置信度：确定。

## 二、直接证据

### 1. 剪贴板包含多位候选人

`logs/clipboard_dump.txt` 当前保存的全页复制文本中，依次出现了大量不同候选人：

- 李泽汉
- 高文昕
- 叶思伟
- 周滨
- 周佳茵
- 李琳
- 魏澎鑫
- 张永康
- 以及更多候选人

这不可能是一份候选人详情。它是被覆盖的底层推荐页与当前全屏详情的混合文本；这里描述的是复制结果，不表示屏幕上存在可见的左右菜单。

文件前部先连续列出推荐列表中的候选人摘要；后部才出现当前详情特有的“教育经历”“工作经历”等结构。程序没有识别两部分的边界，而是对全部文本执行关键词搜索。

### 2. “哈尔滨”误触发可以被完整解释

历史日志显示，关键词“哈尔滨”连续触发了三次。

完整剪贴板中，“哈尔滨”出现在底层推荐页候选人李泽汉的摘要中：

```text
李泽汉
...
哈尔滨信息工程学院 游戏软件开发 大专
高文昕
...
```

切换当前候选人后，底层推荐页仍保留李泽汉，因此每次全页复制仍包含“哈尔滨”。程序执行：

```python
if kw.lower() in text.lower():
    return True
```

于是当前候选人详情即使完全不含“哈尔滨”，也会被判断为命中。

### 3. 其他连续触发同样符合该模式

当前剪贴板中还可以看到：

- “数字媒体”出现在底层推荐页候选人的专业和优势中；
- “广东工贸职业技术学院”出现在底层推荐页候选人的教育摘要中；
- 多个“广东……”和“广西……”院校长期保留在候选人列表中。

这与日志中“数字媒体”“广东工贸职业技术学院”等关键词连续触发的记录一致。

### 4. 文本开头候选人不变不能证明翻页失败

旧版报告曾把连续复制文本开头姓名相同解释为“仍在同一位候选人”。结合现场观察与完整剪贴板结构后，这个推断不成立。

文本总是从被覆盖的底层推荐页开始，所以当前全屏详情已经切换时，剪贴板开头仍可能是同一个推荐页候选人。旧版报告对此判断过强，本修订版予以纠正。

## 三、故障链路

```text
候选人 A 的当前详情含关键词
        ↓
Ctrl+A 复制整个页面
        ↓
关键词命中，转发候选人 A
        ↓
点击详情页恢复焦点，右方向键切换到候选人 B
        ↓
底层推荐页仍保留候选人 A 的摘要
        ↓
再次 Ctrl+A 复制整个页面
        ↓
复制文本仍包含候选人 A 的关键词
        ↓
程序误认为候选人 B 命中并转发候选人 B
        ↓
后续候选人继续误触发
```

## 四、为什么不是布尔状态残留

`view_candidate()` 每轮都会重新创建局部变量：

```python
keyword_hit = False
keyword_hit = detect_keywords()
```

`detect_keywords()` 每次也会重新遍历剪贴板文本，没有保存上一次的命中结果。

`forward_consecutive` 虽然是全局变量，但只用于限制连续转发最多 5 次，不参与关键词是否命中的判断。因此不存在“第一次把某个全局命中开关设为 True，后面一直没清零”的代码路径。

真正被“保留”的是全屏详情覆盖层下面、仍存在于网页文档中的候选人文本，而不是 Python 布尔变量。

## 五、全程序检查发现的问题

### P0：关键词检测范围是整个页面

位置：`select_all_and_copy()`、`detect_keywords()`。

程序注释称“复制详情页文本”，但实际使用网页级 `Ctrl+A`，无法限制在当前全屏详情。这是本次事故的直接根因。

影响：错误转发候选人简历，属于最高优先级。

### P0：没有候选人级防重复或命中归属验证

程序不知道关键词属于页面中的哪一位候选人，也没有当前候选人 ID、姓名或详情区域边界。只要网页文档任意位置包含关键词就会转发当前候选人。

同时，没有“同一候选人一次运行最多转发一次”的硬保护。

### P1：剪贴板没有新鲜度校验

复制前不清空剪贴板，也不写入哨兵值；`Ctrl+C` 失败时会继续读取旧内容。旧内容包含关键词时，也会误触发。

### P1：候选人切换没有验证

`next_candidate()` 只发送一次右方向键并等待 0.5 秒：

```python
pyautogui.press('right')
return safe_wait(0.5)
```

返回 True 只代表没有停止，不代表候选人真的切换成功。虽然这不是本次已确认的主因，但焦点异常时仍可能重复处理同一人。

### P1：窗口识别会误选自身控制台

`bring_edge_foreground()` 只判断窗口标题是否包含 `BOSS` 或 `zhipin`，不校验进程。

日志多次显示所谓的 Edge 窗口其实是：

```text
C:\Users\User\WorkBuddy\...\BOSS直聘刷简历.exe
```

这明显是程序自身窗口。它可能导致键盘事件或复制操作发往错误窗口。

### P1：邮箱检测也可能读取旧剪贴板

程序点击邮箱输入框后执行 `Ctrl+A`、`Ctrl+C`，再用：

```python
if '@' in box_text and '.' in box_text:
```

判断邮箱是否填入。如果复制失败，而旧剪贴板恰好含有邮箱地址，会被误判为输入框已有邮箱。

日志中还出现过把程序控制台提示文字读作邮箱输入框内容的情况，说明焦点问题已经真实发生。

### P2：`--no-forward` 参数无效

`parse_args()` 会解析 `--no-forward`，但 `run()` 和 `get_user_input()` 从不使用该值。用户传入该参数仍可能启用转发。

### P2：转发结果没有真实性校验

点击转发按钮并等待 1–2 秒后，程序直接记录“转发完成”，没有识别成功提示、错误提示或弹窗状态。因此日志中的“转发完成”实际只表示按钮已被点击。

### P2：恢复焦点坐标常量存在混乱

定义了 `RIGHT_CLICK_X/Y`，注释称用于“右键恢复键盘焦点”，但实际从未使用；代码使用的是 `DETAIL_CENTER_X/Y`，并执行左键点击。

这不是当前关键词误判的直接根因，但说明设计、注释和实现已经不一致，容易使后续校准和维护出错。

### P2：计数先于成功处理递增

`total_viewed += 1` 在检测和浏览完成前执行。复制失败、转发异常或浏览中断时，统计值仍可能增加。

### P2：连续转发上限不能阻止错误判断

达到 5 次后 `forward_one_candidate()` 会返回 False，但 `view_candidate()` 仍保留 `keyword_hit = True`，日志仍会显示该候选人命中。程序也不处理转发函数的返回值。

## 六、建议的修复原则

本次不修改代码，仅给出后续设计方向。

1. **只检测当前候选人的全屏详情文本。** 这是必须首先解决的问题。不能再直接对整个网页文档的 `Ctrl+A` 结果全文匹配。
2. **如果暂时只能全页复制，必须可靠提取当前详情区。** 应根据稳定边界切片，并用真实样本验证；仅凭姓名字符串切分容易遇到同名和重复字段，不能草率处理。
3. **记录当前候选人身份。** 至少提取姓名加稳定字段形成指纹，更理想的是使用页面可获得的候选人 ID。
4. **同一身份最多转发一次。** 这是最后一道防止重复邮件的安全闸。
5. **翻页后验证身份变化。** 未变化则恢复焦点并重试，超过次数后安全停止。
6. **剪贴板使用哨兵值。** 复制失败或内容未更新时禁止匹配和转发。
7. **窗口按进程识别。** 只选择 `msedge.exe` 对应窗口，排除自身控制台。
8. **转发前进行二次确认。** 重新读取当前详情区域，确认关键词仍属于当前候选人，再点击转发。

## 七、建议的验收场景

- 候选人 A 命中，候选人 B 不命中，但底层推荐页仍保留 A：只能转发 A。
- 底层推荐页其他候选人含关键词，当前详情不含：不得转发。
- 当前候选人详情含关键词，但底层推荐页摘要不含：应正常转发。
- 连续两个不同候选人均命中：各转发一次。
- 同一候选人被重复打开：不得重复转发。
- 右方向键第一次失效：不得把循环次数误当成候选人已切换。
- `Ctrl+C` 失败：不得使用旧剪贴板进行匹配。
- 控制台标题包含 `BOSS`：不得选中控制台。
- 转发按钮点击后出现失败提示：不得记录为成功。

## 八、最终判断

用户现场观察与完整剪贴板证据相互吻合：右方向键切换可以成功，但程序搜索的是整个网页文档。上一位命中的候选人文本仍存在于被全屏详情覆盖的底层推荐页中，因此关键词也一直留在后续的复制文本里。

本次致命问题的首要修复点应是“将关键词检测严格限定到当前候选人详情”，而不是重置 `keyword_hit`，也不能只靠再次点击屏幕中心解决。
