# BossOCR

BossOCR 是一个面向 Windows 的 BOSS 直聘候选人简历辅助浏览工具。它使用鼠标和键盘模拟完成候选人切换，通过本地 RapidOCR 识别屏幕上当前候选人的可见简历内容；只有关键词经过两次 OCR 精确确认后，程序才允许进入已有的邮件转发流程。

项目不读取网页 DOM，不调用 BOSS 接口，也不使用 Selenium、Playwright、WebDriver、浏览器调试端口或 JavaScript 注入。OCR 和关键词匹配全部在本机完成。

> 使用前请确认你的操作符合 BOSS 直聘服务条款、招聘合规要求和个人信息保护规定。程序包含真实鼠标/键盘自动化，必须先在 `--no-forward` 安全模式下验证。

macOS Chrome 目前仅提供浏览器启动与权限诊断 baseline，**不是完整业务可用版**。请参阅 [macOS Chrome preflight 与人工验收指南](docs/macos-chrome-preflight.md)。

## 下载与运行

推荐从 [GitHub Releases](https://github.com/carolineasu2005-droid/Boss-OCR/releases/latest) 下载 `BossOCR-Windows-x64.zip`。

1. 解压整个 ZIP，不能只复制 `BossOCR.exe`。
2. 在 Microsoft Edge 登录 BOSS 直聘并打开“推荐牛人”页面。
3. 把 Edge 放在主显示器上，不要最小化。
4. 首次测试双击 `安全测试_只检测不转发.bat`。
5. 输入关键词规则；关键词使用英文双引号，规则之间用英文分号 `;` 分隔。
6. 普通交互模式会先询问是否校准“最近没看过”筛选和首位候选人区域，再询问是否校准完整邮件转发点击区域；首次部署建议在 `--no-forward` 下选择 `y` 安全验证。
7. 输入本次运行时间（秒）；留空或输入 `0` 表示持续运行。
8. 按提示依次框选首位候选人卡片、打开筛选、最近没看过、筛选确定四个区域。程序应用筛选并自动打开第一位候选人后，再按提示完成焦点恢复、五个转发点击区域和 OCR 详情正文区域校准。
9. 如果跳过或取消自动筛选校准，按旧流程将鼠标放到第一位候选人卡片上。
10. 观察日志和实际页面，确认筛选、首位归位和 OCR 命中正确后，再考虑使用 `正式运行.bat`。

运行期间：

- `空格`：暂停或继续
- `Esc`：安全停止

Windows SmartScreen 可能提示“未知发布者”，因为当前 EXE 没有商业代码签名证书。请从本仓库 Release 下载并核对 SHA-256 后再运行。

## 工作原理

### 1. 打开当前候选人

程序启动后将标题包含 `BOSS` 或 `zhipin`、且进程名严格为 `msedge.exe` 的窗口置前。普通交互模式可校准候选人筛选和首位候选人区域；启用后，程序会应用“最近没看过”筛选并自动打开第一位候选人详情，不再要求用户预先把鼠标放在首位卡片上。

后续候选人仍使用键盘右方向键切换；每浏览 100 位候选人后仍按 `F5` 刷新。启用自动筛选归位时，刷新成功后会再次应用“最近没看过”并点击首位候选人；未启用时继续使用启动阶段保存的旧首位坐标。这些行为与 OCR 模块相互独立。

#### 可观察的鼠标移动

程序通过 `human_click()` 执行的区域点击默认使用 WindMouse 的 PyAutoGUI backend。距离不足 300 像素时使用一段稳定轨迹；更远时先快速接近预目标点、再稳定收尾，以减轻远距离进入小区域时的终点回拉。每次结束都会强制回到已选定的目标整数坐标，按下和抬起也使用同一坐标。

若 WindMouse 缺失、导入失败或运行时移动异常，程序会记录 warning 并自动回退到原有三次贝塞尔路径，因此不会中断上层流程。WindMouse 依赖固定为 `windmouse==1.0.2`，项目地址为 <https://github.com/AsfhtgkDavid/windmouse>，许可证为 GPL-3.0-only；本仓库不复制或修改其源码，正式分发前须确认发布方式满足该许可证义务。Mac 用户仍须按 [macOS beta 打包与运行](docs/macos-beta-packaging.md) 配置辅助功能、屏幕录制和输入监控等权限。

该移动方式自动覆盖筛选区域、邮件转发区域和转发结束后的焦点恢复点击，不需要为各业务步骤分别配置速度。程序没有 `fast` / `normal` / `slow` 等全局速度档位。未启用自动筛选时，旧首位候选人坐标仍保留原有的直接点击行为。

如需兼容旧版简单直线移动，可增加 `--simple-mouse`。该参数只让 `human_click()` 恢复为一次直线 `moveTo` 后点击，不改变目标区域、点击偏移、业务等待、筛选、转发或键盘流程。

此功能用于提升本地 GUI 操作的可观察性并统一鼠标移动封装，不用于规避验证码、平台风控或自动化检测，也不保证对这些机制产生任何影响。

#### “最近没看过”筛选与首位候选人校准

普通交互模式会询问是否校准自动筛选归位区域。选择 `y` 后依次框选：

1. 首位候选人卡片内部安全区域。
2. “打开筛选”按钮内部安全区域。
3. 筛选面板中的“最近没看过”选项内部安全区域。
4. “筛选确定”按钮内部安全区域。

校准期间只有“打开筛选”区域会被点击，用于展示后两个控件；“最近没看过”“筛选确定”和首位候选人均只框选、不点击。四项全部成功后才会作为整组启用，并且只保存在当前运行内存中。程序随后按 `打开筛选 → 最近没看过 → 筛选确定 → 首位候选人` 的顺序点击；“最近没看过”是筛选面板中的确定项，不是反复开关的 toggle。

任一框选取消、校准异常或校准后关闭面板失败时，本次运行禁用新功能并回退旧首位坐标流程，程序不会因此整体退出。一旦四项校准成功并开始自动应用筛选，任一步点击或等待失败都会安全停止本轮运行，不会在筛选面板可能处于中间状态时盲目切换到旧坐标。

`--auto`、通过 `--keywords` 进入的非交互路径，以及 `--no-batch-filter` 都不会询问或弹出这四个框选层，也不会执行自动筛选导航。`--no-forward` 与该功能相互独立：普通交互模式仍可在禁止真实转发的前提下安全验证筛选和首位归位。没有关键词的普通交互浏览也可以启用该功能。

用户设置的运行时间从首位候选人打开、焦点/转发/OCR 等启动准备完成后才开始计算；四区域框选和首次筛选归位不消耗运行时间。

### 2. 首次检测时拖框校准

第一位候选人详情已经打开后，程序才显示 Tk 半透明拖框。用户框选当前候选人的详情正文区域，程序记录：

```text
left, top, width, height
```

校准区域只保存在本次运行的内存中，程序重启后需要重新框选。校准完成时会通过 MSS 对同一区域截图，并保存预览到：

```text
logs/ocr_calibration_preview.png
```

Windows 下程序启用 Per-Monitor V2 DPI 感知，并将 Tk 的界面坐标换算成 MSS 使用的物理像素坐标。当前第一版只支持主显示器；多显示器混合 DPI 环境应把 Edge 移到主显示器。

#### 完整邮件转发点击区域

有关键词的普通交互模式会在启动时询问是否校准完整邮件转发点击区域。选择 `y` 后，程序会先打开第一位候选人详情页，然后按顺序引导框选：

1. 候选人详情页的焦点恢复空白区域。
2. 右上角“转发牛人”图标内部安全区域。
3. 转发弹窗左侧“邮件转发” Tab 内部安全区域。
4. 邮箱输入框内部安全区域。
5. “最近联系”中第一个邮箱标签内部安全区域。
6. 弹窗右下角“转发”按钮内部安全区域。

程序仅使用前两个新框选区域打开转发弹窗并切换到邮件 Tab；邮箱输入框、最近联系和转发按钮在校准期间只框选。校准期间绝不点击“转发”按钮，完成后使用 Esc 关闭弹窗。这个流程不识别页面状态；如果界面与提示不符，应按 Esc 取消。

六个区域都只保存在本次运行的内存中，不写入配置文件，程序重启后需要重新选择。真实转发时，程序从各运行期区域内取点点击；转发处理函数退出前仍会从焦点恢复区域内取点，恢复详情页键盘焦点。

直接回车、选择 `n`、任意一项按 Esc 取消或校准失败时，五个新转发区域会作为整组回退到由现有固定坐标推导的默认小区域。焦点恢复区域继续独立回退到：

```text
X:400-500, Y:350-400
```

`--auto` 模式不会询问、不会弹出任何转发框选层，也不会执行校准导航点击，始终使用全部默认区域。当前不提供非交互校准参数。校准结果是绝对屏幕坐标，校准后不要移动、缩放 Edge 窗口或改变浏览器缩放比例。

### 3. 局部屏幕截图

OCR 只截取拖框区域内当前真实可见的像素，不会截取浏览器整页，也不会读取被详情遮挡的底层候选人列表文本。这解决了旧方案通过 `Ctrl+A` / `Ctrl+C` 复制整个网页时，其他候选人的关键词可能误触发当前候选人的问题。

截图后端使用 `mss`，像素数据直接在内存中交给 RapidOCR。除校准预览外，不需要把每一屏保存到磁盘。

### 4. RapidOCR 本地识别

程序启动时创建一个 RapidOCR 实例，并在整个运行期间复用。推理由 ONNX Runtime CPU 后端执行，默认包含三个本地模型：

- 文字检测模型
- 文字方向分类模型
- 中文/英文文字识别模型

模型随 Windows Release 一起分发，不需要把候选人截图上传到外部 OCR 服务。

每个 OCR 文字框包含文字、位置和置信度。低于 `0.85` 的文字框不会参与关键词匹配。剩余文字框按自适应行高从上到下、同行从左到右排序。

### 5. 标准化后的关键词规则匹配

OCR 文本与用户关键词执行相同的标准化：

- Unicode NFKC 全角/半角归一化
- 英文字母转小写
- 删除换行和布局空白

每个带英文双引号的关键词只执行标准化后的精确子串匹配：

```text
normalized_keyword in normalized_ocr_text
```

同一条规则内支持 `any(...)`、`not`、`and` 和 `or`，优先级为 `not` > `and` > `or`；分号分隔的多条规则只要任一条命中即可。

- `any("A","B","C")` 表示 A、B、C 任意一个关键词命中即可。
- `any(...)` 是一个完整原子条件，不会展开成外层 OR 分支，因此多个 any 之间的 `and` 约束会被保留。
- `not` 只能修饰一个带英文双引号的关键词或一个 `any(...)`，表示文本中不能包含对应关键词。
- 每条规则必须包含正向条件。
- 每个 OR 分支都必须至少包含一个正向条件，纯 not 分支会被拒绝。
- `or` 继续用于连接多个完整筛选分支，不会被 `any(...)` 取代。

合法示例：

```text
"短剧" and not "销售"
"AIGC" and not "运营"
"A" or not "B" and "C"
"A" and not "B" or "C"
any("魔方","九州") and any("短剧","漫剧")
"魔方" and not any("投放","消耗")
any("A","B") or "C"
```

例如：

```text
any("魔方","九州") and any("短剧","漫剧")
```

要求两个条件组分别至少命中一项。只有 `魔方` 或只有 `短剧` 都不会命中，`魔方 短剧` 才会命中。`any(...)` 不会被展开为多个顶层 OR 分支。

```text
"魔方" and not any("投放","消耗")
```

`魔方 投放` 不命中，`魔方 剪辑` 命中。`not any("投放","消耗")` 表示“投放”和“消耗”任意一个出现都会排除。

`any("A")` 合法且语义等价于 `"A"`，主要方便配置生成器或临时删减配置，但人工配置时不推荐这种冗余写法。

其中：

```text
"A" or not "B" and "C"
```

等价于：

```text
"A" or ((not "B") and "C")
```

第二个 OR 分支包含正向关键词 `"C"`，因此合法。以下规则包含纯 not 分支，必须拒绝：

```text
not "B"
not "B" or "A"
"A" or not "B"
```

`any(...)` 内只允许一个或多个由英文逗号分隔的英文双引号关键词。同一个 any 内的语义重复项会被拒绝，不会自动去重；例如 `any("A","A")` 非法。以下写法均不支持：

```text
any(any("A","B"),"C")
any("A" and "B","C")
("A" or "B") and "C"
any(A,B,C)
any('A','B')
```

除 `any(...)` 自身的参数括号外，程序仍不支持完整括号表达式、数值比较或 `not` 修饰 AND/OR 组合，也不做编辑距离、相似字替换、拼音匹配、同义词推断、语义匹配或大模型判断。错误格式会明确报错，不会静默忽略、自动改写或把操作符当作普通关键词。

### 6. 最多 8 屏有序扫描

每位候选人的第一屏直接截图识别。未命中时，程序始终向下滚动约 5–7 格，等待页面稳定约 `0.6` 秒后继续截图，最多扫描 8 屏。

```text
第一屏 OCR
    ↓ 未命中
向下滚动并等待稳定
    ↓
第二屏 OCR
    ↓
……最多第八屏
```

OCR 扫描耗时计入原有每位候选人 12–18 秒停留预算；扫描结束后不会再完整等待一轮 12–18 秒。OCR 等待和滚动期间仍响应空格暂停与 Esc 停止。

### 7. 同一规则二次确认

首次命中规则后，程序不会立即转发。它保持当前滚动位置，等待约 `0.7` 秒，对同一区域再次截图和 OCR。两次都命中同一整条规则（包括全部正向和 not 排除条件），才产生 `confirmed_match=True`。

以下情况全部禁止转发：

- OCR 引擎初始化失败
- 用户取消或未完成拖框
- MSS 截图失败
- OCR 调用异常
- OCR 返回空结果
- 只有低置信度文字命中
- 第二次 OCR 未确认同一整条规则

失败时不会回退到网页全选复制方案。

### 8. 转发安全门

`--no-forward` 模式仍会完整执行拖框、OCR、滚动、二次确认和日志记录，但即使命中也不会调用邮件转发函数。这是首次部署和关键词验收必须使用的模式。普通交互方式下，只要设置了关键词，即使启用 `--no-forward` 也可以验证完整转发校准流程。校准导航只点击转发入口和邮件 Tab，绝不点击转发按钮；后续 OCR 命中也不会执行真实转发或转发后的焦点恢复点击。

正式模式下，只有二次确认命中才会调用原有 `forward_one_candidate()`。转发入口、邮件 Tab、邮箱输入框、最近联系人和转发按钮均使用本次运行的校准区域或默认区域；转发处理的所有退出路径仍使用 Next-1 的焦点恢复区域。邮箱输入框内部保留剪贴板检查逻辑，但候选人关键词检测完全不使用剪贴板。

真实转发验证只能使用测试账号和测试邮箱，且必须小规模、全程人工监控。发现页面错位、弹窗未打开或任何误点时立即按 Esc 停止。

## 命令行用法

安全检测，不发送邮件：

```powershell
BossOCR.exe --keywords '"Python"; "短剧"' --no-forward --auto
```

源码环境：

```powershell
venv\Scripts\python.exe simple_brush.py --keywords '"Python"; "短剧"' --no-forward --auto
```

正式模式示例：

```powershell
BossOCR.exe --keywords 'any("Python","剪映") and not any("销售","投放")' --email "backup@example.com" --auto
```

| 参数 | 作用 |
| --- | --- |
| `--keywords` | 关键词规则；支持 `any(...)`、`not`、`and`、`or`，多条规则用英文分号分隔 |
| `--email` | 最近联系人没有邮箱时使用的备用邮箱 |
| `--no-forward` | 完整执行 OCR，但禁止真实邮件转发 |
| `--no-batch-filter` | 禁用“最近没看过”自动筛选归位，不询问或框选，使用旧首位坐标流程 |
| `--simple-mouse` | 区域点击使用旧版简单直线鼠标移动；不改变点击目标和业务流程 |
| `--duration-seconds` | 本次运行秒数；省略或设为 `0` 表示持续运行 |
| `--auto` | 跳过交互输入、自动筛选归位校准和全部转发区域校准，不执行校准导航，使用命令行参数及旧首位/默认区域 |

不提供关键词时，OCR 和自动转发均禁用，只执行候选人浏览。
普通交互模式无论是否提供关键词，都可选择校准自动筛选归位；有关键词时还会询问是否校准完整转发点击区域，无关键词时不询问转发校准。使用 `--auto` 时可通过 `--duration-seconds 3600` 设置自动停止时间；非法值会阻止程序启动。

关键词规则示例：

```text
"剪映" and "信息流";
"短剧" and not "销售";
any("魔方","九州") and any("短剧","漫剧") and not any("投放","消耗");
"A" or not "B" and "C"
```

`any(...)` 表示组内任一关键词命中，`not` 要求排除指定关键词或 any 组，`and` 要求同一完整分支内的条件全部满足，`or` 连接多个完整筛选分支；混用时优先级为 `not` > `and` > `or`。每个 OR 分支都必须有正向条件。除 any 自身的参数括号外，不支持完整括号表达式、数值比较或组合取反。旧配置 `Python;短剧` 必须手动迁移为 `"Python";"短剧"`，程序不会静默改写错误格式。

## 从源码运行

### 环境要求

- Windows 10/11 x64
- Python 3.10 或更高版本（推荐 Python 3.11 x64）
- Microsoft Edge
- Edge 位于主显示器

### 安装

双击：

```text
setup.bat
```

或手动执行：

```powershell
py -3.11 -m venv venv
venv\Scripts\python.exe -m pip install -r requirements.txt -r requirements-ocr.txt
```

启动：

```powershell
venv\Scripts\python.exe simple_brush.py --keywords '"测试关键词"' --no-forward --auto
```

运行测试：

```powershell
venv\Scripts\python.exe -m unittest discover -s tests -v
```

### Windows 发布构建

在 Windows 10/11 x64 上运行：

```text
build-windows.bat
```

脚本会安装构建依赖、运行全量测试、使用 `BossOCR.spec` 生成控制台版 one-dir 应用，并以 `--no-forward --auto` 做安全冒烟测试。成功后发布压缩包位于：

```text
release\BossOCR-Windows-x64.zip
```

`build/`、`dist/`、`release/`、虚拟环境、日志和缓存均为本地产物，不应提交到 Git。

## 代码结构

```text
simple_brush.py      主循环、窗口控制、候选人切换、暂停/停止和邮件转发安全门
ocr_calibration.py   Tk 拖框、Windows DPI 坐标换算、校准预览
ocr_detector.py      MSS 截图、RapidOCR 适配、8 屏扫描和二次确认
ocr_text.py          OCR 文字框排序、文本标准化和精确关键词匹配
tests/               校准、OCR、失败关闭、no-forward 和主流程测试
```

## 日志与隐私

运行数据统一保存在 `logs/`，该目录已被 Git 忽略：

```text
logs/simple_brush.log
logs/ocr_calibration_preview.png
logs/ocr_hits/
```

日志至少记录 OCR 初始化、校准区域、扫描序号、OCR 耗时、文字框数量、命中关键词、二次确认、错误和安全跳过原因。

校准预览和命中截图可能包含候选人个人信息。不要提交、上传、分享或长期保留 `logs/`，并按适用的数据保护规则定期清理。

## 当前限制

- 第一版只支持主显示器拖框；混合 DPI 多显示器尚未支持。
- OCR 只识别当前屏幕可见内容，遮挡、动画、加载状态和字体清晰度会影响结果。
- `not` 仍使用标准化后的精确子串匹配；过短的排除词（例如 `"A"`）可能造成大范围误排除。
- `any(...)` 内部仍是标准化后的精确子串匹配，不支持嵌套 any 或通用括号表达式。
- 每位候选人固定最多扫描 8 屏，不判断简历视觉底部。
- 增强鼠标移动会为每次区域点击增加约 `0.20~0.75` 秒移动时间；长流程的总耗时会相应增加。
- 贝塞尔移动只覆盖经 `human_click()` 执行的区域点击；旧首位候选人直接点击、滚轮和键盘操作保持原行为。
- 转发点击校准是绝对屏幕坐标，不会跟随窗口；校准后移动窗口、改变系统或浏览器缩放、切换分辨率或页面布局，都需要重启并重新校准。
- 自动筛选归位同样使用当前运行期的绝对屏幕区域；窗口位置、系统或浏览器缩放、分辨率或页面布局变化后需要重启并重新校准。
- 自动筛选不读取按钮文字、不检测筛选是否生效，也不判断列表或详情页是否加载完成；慢网络、空列表、骨架屏或页面布局异常可能导致安全停止或误点。
- 校准结束时通过 Esc 最佳努力关闭筛选面板；Esc 能否在当前 BOSS 页面稳定关闭面板，以及筛选确定后首位卡片是否仍位于校准区域，需要在实际页面人工确认。
- 跳过、取消或校准失败时会回退到现有默认坐标区域；默认值基于 1920×1080 Windows Edge 布局，其他布局必须先人工验证。
- 完整业务流程仍仅支持 Windows + Microsoft Edge；macOS Chrome 当前仅支持 [preflight baseline](docs/macos-chrome-preflight.md)，不会进入业务循环。
- EXE 当前未进行商业代码签名。
- 自动化工具不能替代人工招聘判断；正式启用前应使用 `--no-forward` 人工核对足够样本。

## 安全建议

1. 新电脑、新分辨率或页面布局变化后，先以普通交互方式使用 `--no-forward`，验证四个筛选归位区域、OCR 和全部转发点击区域框选。
2. 人工核对关键词在首屏、后续屏、无关键词、not 排除词出现和二次确认失败等场景；排除词应尽量具体。
3. 确认 `logs/` 中没有 OCR 或截图错误。
4. 仅使用测试账号和测试邮箱小规模验证真实转发，并在验证期间全程人工监控。
5. 发现误点或异常时立即按 Esc。
